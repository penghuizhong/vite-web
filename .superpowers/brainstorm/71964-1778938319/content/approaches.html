移植方案对比

三种实现浮动聊天面板的方式，各有取舍

A

全局浮动组件 (推荐)

在 AppLayout 中挂载 GlobalChatShell，所有页面可用。侧边栏点击 → 打开浮动面板 + 路由跳转同步。桌面端 500×812px，手机端 100vw×80vh。

优点

  • 任何页面都能唤起聊天
  • 与侧边栏会话列表无缝联动
  • old-web 代码复用度最高

缺点

  • 需要废弃当前 ChatPage 路由页
  • 需重构 chatStore 与 UI 状态对接
B

路由页 + 浮动模式切换

保留 /chat 路由页，但增加浮动模式。侧边栏点击时，不跳转路由，而是从任意页面唤起浮动面板。

优点

  • 保留现有路由结构
  • 用户可全屏聊天也可浮动

缺点

  • 两种模式需维护两套状态
  • 逻辑复杂，容易不同步
C

纯路由页面改造

不采用浮动面板，将 old-web 的 UI 风格（圆角输入框、消息气泡、动画）移植到现有 ChatPage 中。

优点

  • 改动最小，不动架构
  • 路由逻辑清晰

缺点

  • 丢失"随时可聊"的浮动体验
  • 无法在其他页面唤起聊天
